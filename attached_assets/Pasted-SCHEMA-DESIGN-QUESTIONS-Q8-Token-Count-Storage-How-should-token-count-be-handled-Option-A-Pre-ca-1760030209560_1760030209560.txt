SCHEMA DESIGN QUESTIONS
Q8: Token Count Storage
How should token_count be handled?

Option A: Pre-calculated and stored in metadata (static, needs manual update if document changes)
Option B: Calculated dynamically at retrieval time (accurate but adds processing)
Option C: Estimated from character count formula (fast approximation)

My recommendation: Option A - Pre-calculated and stored. Documents change infrequently, token accuracy matters for budget.

Q9: Document Versioning
Since these seed documents will evolve over time (refinements, additions):

Option A: Simple updates (new version overwrites old, no history kept)
Option B: Versioning system (keep previous versions, track changes)

My recommendation: Option A for Phase 1 - Simple updates, no versioning complexity. Add versioning in Phase 2 if needed.

Q10: Agent Recommendation Field
Some documents have agent_recommendation: "Sarah" or "Mathew".

Should this be stored as plain text (agent_recommendation TEXT)?
Or should there be an agents lookup table (agent_id INTEGER FOREIGN KEY)?

My recommendation: Plain text for Phase 1 - Simpler, fewer joins. Normalize to lookup table in Phase 2 if agents become more complex.
